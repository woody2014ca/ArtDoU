**文档日期**：2026-03-01

## 一、项目与架构

| 组件 | 技术/平台 | 地址/说明 |
|------|-----------|-----------|
| 前端（老师/家长管理端） | React + Vite，部署于 Vercel | 域名：www.kunlunfo.com |

## 二、当前现象（用户可见）

1. **后端健康检查**
   - 访问 **https://artdou-production.up.railway.app/health** 返回 **502 Bad Gateway**，页面提示 “Application failed to respond”。

2. **前端登录**
   - 访问 **www.kunlunfo.com**，输入密码点击登录后，页面提示 **「网络错误，请检查后端是否已启动或 API 地址」**。

3. **Railway 控制台内表现**
   - **Build Logs**：构建成功（约 20 秒，各步骤绿色勾选）。
   - **Deployments**：存在 **Active** 部署（如 0017a9f5），状态为在线。
   - **Deploy Logs**（运行时）：可见 “Starting Container” → “API listening on 8080” → “MongoDB connected”，无后续报错截断。
   - **HTTP Logs**：对该服务的请求（如 `GET /health`、`OPTIONS /auth/login`）**全部返回 HTTP 502**。
   - **Health Check Path**：已按建议留空；Redeploy 后上述现象未改变。

即：**构建与部署状态正常，运行时日志显示进程已启动并连上数据库，但所有对外 HTTP 请求均得到 502。**

## 三、已做过的变更与尝试（时间线摘要）

| 阶段 | 内容 | 结果 |
|------|------|------|
| 1 | 后端先启动 HTTP 再后台连 MongoDB，避免连接挂起导致 502 | 出现 “API listening on 8080”，但仍有 502 |
| 2 | 连 MongoDB Atlas 时出现 **SSL alert 80**（TLS 握手错误） | 多次尝试：换 Node 镜像、IPv4、TLS 选项、SRV 解析、驱动升级等，均未消除 Atlas 连接报错 |
| 3 | 改用 **Railway 自带 MongoDB**（同项目内），不再连 Atlas | Deploy Logs 出现 “MongoDB connected”，SSL 问题消失 |
| 4 | 曾出现 **ENOTFOUND mongodb.railway.internal** | 确认为 MongoDB 与 ArtDoU 不在同一 Railway 项目；调整为同项目后解决 |
| 5 | 曾出现部署为 **Removed** 导致无实例接流量 | 用户进行 Redeploy，得到新的 **Active** 部署 |
| 6 | 前端 **VITE_API_URL** 在 Vercel（zhifou 项目）中设为 `https://artdou-production.up.railway.app` | 配置已存在；因后端持续 502，前端仍报网络错误 |
| 7 | Railway 上 **Health Check Path** 留空并 **Redeploy** | 两个网页报错依旧（后端 502，前端网络错误） |

## 四、问题归纳（供审计与排查）

- **现象**：后端服务在 Railway 上构建成功、部署为 Active、Deploy Logs 显示进程已监听 8080 且 MongoDB 已连接，但所有到达该服务的 HTTP 请求（含 `/health`）均返回 **502**；前端因无法得到后端正常响应而报「网络错误」。
- **矛盾点**：同一部署下，**运行时日志** 显示服务已就绪，**HTTP 访问结果** 却一致为 502，说明请求未由该进程正常响应（或未到达该进程）。
- **可能方向**（供外部审计参考，不排除其他原因）：
  1. **路由/代理**：Railway 将公网请求转发到该服务时，端口、协议或实例选择有误，导致请求未到达实际监听 8080 的进程。
  2. **健康检查与实例状态**：若平台存在内部健康检查（即便 Health Check Path 留空），检查失败可能导致流量未打到当前 Active 部署，或实例被标记为不健康从而返回 502。
  3. **进程生命周期**：进程在 “MongoDB connected” 之后、请求到达前退出，或仅在启动阶段短暂监听，需结合更完整的 Deploy Logs 与平台指标（如重启次数、OOM）进一步确认。
  4. **多实例/多部署**：若存在多个部署或副本，流量可能被分配到已移除或未就绪的实例，需在平台侧确认当前对外流量实际指向的部署与实例。

## 六、建议的后续排查（给运维/审计）

1. 在 Railway 查看该服务的 **Settings**：确认 **Port**、**Public Networking**、**Health Check**（若有）与文档或平台默认值是否一致；确认当前 **Active** 部署是否为该服务唯一接收流量的部署。
2. 查看该 Active 部署的 **完整 Deploy Logs**（从 “Starting Container” 到最近时间）：确认在 “MongoDB connected” 之后是否有异常、重启或退出记录。
3. 查看 Railway 项目或服务级别的 **Metrics / 实例状态**：是否有频繁重启、OOM 或 Unhealthy 等。
4. 若有条件，在 Railway 使用 **Shell / Exec** 进入运行中容器，在容器内执行 `curl localhost:8080/health`（或实际 `PORT`）验证进程在本机是否响应。

## 七、审计结论与修复（L4 绑定）

**根因（外部审计结论）**：问题在 **L4 TCP 绑定**，不在 L7（Atlas TLS、Health Check、Redeploy 等）。进程曾使用 `process.env.PORT || 3001`，在 Railway 未正确注入或生效时监听在 **127.0.0.1:3001**，而非 **0.0.0.0:8080**，导致 Railway 代理无法将公网请求转发到容器内进程，所有请求返回 502。

**唯一正确验证方式（Railway Shell）**：

1. Railway → Service → Deploy → **Open Shell**
2. 执行：`echo $PORT` → 应为 `8080`
3. 执行：`ss -lntp` 或 `netstat -tulpn`  
   - **错误**：`LISTEN ... 127.0.0.1:3001`  
   - **正确**：`LISTEN ... 0.0.0.0:8080`

**代码修复（已落实）**：

- **PORT**：`const PORT = Number(process.env.PORT) || 8080`（不再 fallback 到 3001）；无 PORT 时仅 `console.warn`。
- **强制监听 0.0.0.0**：`app.listen(PORT, '0.0.0.0', ...)`。
- `/health` 返回 `res.status(200).json({ status: 'ok', ok: true })`。

**部署军规**：① 永远 listen 0.0.0.0；② 部署后用 Railway Shell 验证监听端口。

## 八、若仍无法登录（请按顺序做）

1. **在 Railway 显式设置 PORT**
   - ArtDoU 服务 → **Variables** → 添加 `PORT` = `8080`（若已有则确保为 8080）→ 保存后触发 Redeploy。

2. **看 Deploy Logs 是否出现这两行**
   - `[ArtDoU] process starting, PORT from env: 8080`
   - `✅ API listening on 8080`
   - 若**没有**第一行：进程可能在 import 阶段就崩了（看是否有报错栈）。
   - 若**没有**第二行：在 listen 之前崩了（看中间是否有报错）。

3. **用 Shell 确认监听（审计建议）**
   - Railway → 该次部署 → **Open Shell**。
   - 执行：`echo $PORT`（应为 8080）。
   - 执行：`ss -lntp` 或 `netstat -tulpn`。
   - 必须看到 **0.0.0.0:8080**，不能是 127.0.0.1:3001。

4. **确认公网端口**
   - 服务 **Settings** → **Networking**（或 Port）：内部端口应为 **8080**（或留空由 PORT 决定）。

5. **再测 /health**
   - 浏览器打开：https://artdou-production.up.railway.app/health  
   - 若返回 `{"status":"ok","ok":true}`，再在 www.kunlunfo.com 用密码登录；若仍 502，把上面 2、3、4 的结果发给运维或继续排查代理配置。
